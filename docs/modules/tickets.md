# Module: Tickets

## Overview

The ticket system handles support requests that need tracking over time. Tickets can be created from chat handovers, inbound email, the widget, or manually by agents.

---

## Components

- **Controller**: `TicketController` (`app/Http/Controllers/Api/TicketController.php`)
- **Service**: `TicketService` (`app/Services/TicketService.php`) — all side-effects
- **Policy**: `TicketPolicy` — view, update, reply, escalate, assign, delete
- **Public controller**: `PublicTicketController` — customer-facing ticket view (no auth)
- **Frontend**: `pages/dashboards/TicketDetails.tsx`, `pages/customer/PublicTicketPage.tsx`
- **Resources**: `TicketResource`, `TicketMessage` (raw model)
- **Form Requests**: `Ticket/StoreTicketRequest`, `UpdateTicketRequest`, `UpdateStatusRequest`, `AssignTicketRequest`, `EscalateTicketRequest`, `ReplyTicketRequest`

---

## Ticket Statuses

```
open ──── assigned ──→ in_progress ──→ waiting ──→ resolved ──→ closed
 │                          │
 │                          │── escalate ──→ (new agent, in_progress)
 │
 └── pending  (customer waiting for more info)
```

Enum: `App\Enums\TicketStatus` — `open`, `in_progress`, `waiting`, `resolved`, `closed`, `pending`

When status is set to `resolved`, `resolved_at` timestamp is automatically set.

---

## Ticket Priorities

Enum: `App\Enums\TicketPriority` — `low`, `medium` (default), `high`, `urgent`

---

## Ticket Number Format

Auto-generated by `Ticket::booted()` hook after insert:
```
TKT-{YEAR}-{00001}  (e.g. TKT-2026-00042)
```

---

## Access Token (Public View)

Each ticket gets a random 48-char `access_token` set in `booted()`. Public URL:
```
{FRONTEND_URL}/ticket/{access_token}
```

Customers can view and reply to their ticket without logging in.

## TicketService Methods

### `create(array $data, ?User $actor): Ticket`

Side effects in order:
1. Creates `Ticket` record
2. Refreshes to get `ticket_number` + `access_token` from booted hook
3. If `chat_id` is set: creates system chat message + broadcasts `MessageSent`
4. Creates initial `TicketMessage` (from customer)
5. Sends `TicketCreatedMail` to customer (with public ticket URL)
6. Sends `NewTicketMail` to all project agents + owner
7. Logs to `activity_logs`
8. Creates Frubix CRM lead if Frubix integration is enabled on the project

### `assign(Ticket $ticket, ?int $agentId): Ticket`

- Updates `assigned_to` and `status` (`in_progress` if assigned, `open` if unassigned)
- Creates system `TicketMessage`: "Assigned to {agent}", "Unassigned", or "Reassigned from X to Y"

### `escalate(Ticket $ticket, int $agentId, ?string $reason): Ticket`

- Updates `assigned_to` to new agent, `status` to `in_progress`
- Creates system `TicketMessage` with escalation note

### `updateStatus(Ticket $ticket, string $newStatus): Ticket`

- Updates status, sets `resolved_at` if status = `resolved`
- Creates system `TicketMessage`: "Status changed from X to Y"

### `reply(Ticket $ticket, string $content, User $agent, bool $sendEmail): TicketMessage`

- Creates `TicketMessage` from agent
- If `send_email=true`, sends raw email via `sendEmailReply()`

## Inbound Email → Ticket

`InboundEmailController::handle` — called by email provider webhook.

Flow:
1. Email provider (Postmark, Mailgun, etc.) posts to `POST /api/email/inbound`
2. Controller parses sender, subject, body
3. Creates or updates a ticket based on email thread
4. Sends reply notification

## Authorization

| Action | Who can do it |
|--------|--------------|
| View ticket | Assigned agent, any project member, project owner, superadmin |
| Update (subject/desc/priority) | Same as view |
| Reply | Same as view |
| Escalate | Same as view |
| Assign | Project member, project owner, superadmin (NOT just assigned agent) |
| Take (self-assign) | Any project member (only if ticket is unassigned + open/waiting) |
| Delete | Project owner, superadmin only |

## Key Caveats

1. `TicketController::store` returns `$ticket` raw (not wrapped in `TicketResource`) — see `issues.md` (M4).
2. Frubix integration is checked inside `TicketService::create` — if the integration is misconfigured, the lead creation failure is logged but does NOT block ticket creation.
3. The `take` action uses `DB::transaction` with `lockForUpdate()` to prevent race conditions when multiple agents try to take the same ticket simultaneously.
4. Email replies use `Mail::raw()` not a Mailable class — the email has no template, just plain text.
